Full Stack Agentic Development

A spec-driven, agentic-first methodology where small cross-functional pods use markdown as a common language to direct teams of AI agents — collapsing cycle times from weeks to hours.

01 — What is FSAD

The Paradigm Shift

FSAD replaces the traditional handoff-heavy SDLC with a tight feedback loop where humans define what and why in markdown — and AI agents handle the how.

🎯

Spec-Driven, Not Code-Driven

The specification — written in markdown — is the primary artifact. It's a living document that evolves in real-time during collaborative sessions. Code is a compilation target, not the source of truth.

🤝

Cross-Functional Pods

Small teams of 2–4 (PM + UX + Engineer, or PM + 2 Engineers) sit together — physically or virtually — and co-author the spec in real-time. No handoffs. No waiting for the next sprint.

📝

Markdown as Common Language

Markdown is lightweight, version-controlled, diff-friendly, and — critically — the native instruction format for AI agents. Every role can read and write it. It bridges the gap between human intent and machine execution.

🤖

Agent-Delegated Execution

Once the spec is validated, teams of AI agents — running in parallel via tools like Claude Code, Codex, or Gemini CLI — implement the code. The human role shifts from writing code to reviewing, steering, and verifying agent output.

Models forget. Hooks don't. The Hooks principle — see Claude Best Practices › Hooks

Phase 1 — Intent & Discovery

The PM and UX designer define the problem space and user outcomes. They collaborate — often with an AI research assistant — to surface user needs, competitive landscape, and constraints. The output is a concise problem statement in markdown that frames what we're solving and for whom. No solutions yet.

problem.mduser-stories.mdconstraints.md

Phase 2 — Live Spec Session

The entire pod convenes in a real-time collaborative session. The PM contributes business requirements, UX contributes interaction patterns and design decisions, and the Engineer contributes technical feasibility and architecture. They co-author the spec in markdown together — design decisions are embedded inline, not deferred to a separate Figma-to-Jira handoff. This is where FSAD breaks the handoff chain.

spec.mddesign-decisions.mdAGENTS.md

Phase 3 — Implementation Planning & Task Decomposition

The Engineer feeds the spec to an AI agent in "plan mode" (read-only). The agent analyzes the codebase, generates an implementation plan, and decomposes work into discrete, parallelizable tasks. The human reviews the plan against the spec — catching hallucinations, missed edge cases, and architectural misalignment before any code is written.

plan.mdtasks/*.mdtest-plan.md

Phase 4 — Parallel Agent Execution

Multiple AI agents execute tasks simultaneously in isolated git worktrees. Each agent works against the spec as its source of truth. A Coordinator agent (or human) routes tasks, and a Verifier agent checks each output against the spec's acceptance criteria. The engineer monitors, steers, and intervenes only when agents get stuck or drift.

src/**tests/**agent-logs/

Phase 5 — Human Review & Ship

The full pod reviews the implementation. The Engineer checks code quality, architecture adherence, and test coverage. UX validates the interface against design decisions in the spec. The PM validates against user stories and acceptance criteria. Reviews focus on spec compliance, not syntax — because agents produce consistent code from living specs. Then ship.

The Markdown is the Meeting

In FSAD, the spec.md file replaces most of the ceremony of traditional development — the Jira tickets, the Figma handoffs, the sprint planning debates, the slack threads asking "what did we decide?"

PM contributes user intent, business constraints, success metrics, and acceptance criteria directly into the spec.

UX Designer contributes interaction patterns, component decisions, layout descriptions, and accessibility requirements — inline, not as a separate Figma file.

Engineer contributes technical constraints, data models, API contracts, and architecture decisions — and validates feasibility in real-time.

AI Agents consume the spec and execute implementation in parallel — the "workforce" that the pod directs.

06 — Why FSAD Wins

The Advantages

Cycle Time Compression

Eliminating handoffs and delegating implementation to parallel agents compresses 2–4 week sprints into 1–3 days.

No Handoff Latency

No more Figma → Jira → Slack → Code chain. Design decisions reach code without translation loss.

Parallel Execution

Multiple agents in isolated worktrees implement different parts of the spec simultaneously.

Single Source of Truth

spec.md replaces the constellation of Jira, Confluence, Figma, Slack threads, and meeting notes.

Knowledge Compound

Better specs produce better agent output, which produces cleaner codebases, which produces even better agent output. A virtuous cycle.

Democratized Contribution

PMs and designers contribute directly to the artifact that drives implementation. No code required.

The Bottom Line

FSAD is not post-Agile.
It's Agile fulfilled.

The Agile Manifesto valued "individuals and interactions over processes and tools" and "working software over comprehensive documentation." FSAD delivers on both — by making the spec a live interaction between individuals, and by compiling that spec into working software through AI agents in hours, not weeks.

03 — Pod Anti-Patterns

Compositions That Don't Work

🚫

The Solo Hero

One engineer doing everything. No second perspective means spec blind spots go unchallenged. Agents amplify mistakes at scale.

🚫

The Oversized Pod

5+ humans on a single spec. Communication overhead grows quadratically. Split into two pods with a shared API contract.

🚫

The Handoff Pod

PM writes spec → throws to designer → throws to engineer. Same sequential process. If you aren't co-authoring, it's not FSAD.

🚫

The Agent-Only Pod

Fully autonomous agents, no human review. Agent reliability is non-deterministic. Every pod requires human validation gates.

🚫

The Part-Time Pod

Members at 20% allocation across multiple pods. FSAD's speed comes from real-time collaboration. Async stalls the spec session.

🚫

The Duplicate Pod

Two engineers with identical skills. The second must own a distinct domain — otherwise use a Triad and let agents parallelize.

The Principle

Smallest Effective Pod.
Maximum Agent Leverage.

The right pod is the smallest group of humans who can make complete decisions about a feature without external dependencies — then direct an arbitrarily large fleet of agents to execute in parallel.

04 — Inside the Commit-Changes Skill

/commit-changes

The counterpart to /add-task. When a batch of tasks is done, this skill writes the CHANGELOG entry, bumps the version, and commits — in the exact sequence that keeps the artifact coherent.

Find the shipping set

Completed, but not yet in CHANGELOG

The skill reads markdown/to do/todo.md for items marked [x], then cross-references CHANGELOG.md to filter out anything already shipped. If the set is ambiguous — multiple newly-completed tasks, only some meant to go — it asks the user before writing a single line.

## Workflow Orchestration

### 1. Plan Mode Default
- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately — don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

### 2. Subagent Strategy
- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

### 3. Self-Improvement Loop
- After ANY correction from the user: update `tasks/lessons.md` with the pattern
- Write rules for yourself that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops
- Review lessons at session start for relevant project

### 4. Verification Before Done
- Never mark a task complete without proving it works
- Diff behavior between main and your changes when relevant
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness

### 5. Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes — don't over-engineer
- Challenge your own work before presenting it

### 6. Autonomous Bug Fixing
- When given a bug report: just fix it. Don't ask for hand-holding
- Point at logs, errors, failing tests — then resolve them
- Zero context switching required from the user
- Go fix failing CI tests without being told how

## Task Management

1. **Plan First**: Write plan to `tasks/todo.md` with checkable items
2. **Verify Plan**: Check in before starting implementation
3. **Track Progress**: Mark items complete as you go
4. **Explain Changes**: High-level summary at each step
5. **Document Results**: Add review section to `tasks/todo.md`
6. **Capture Lessons**: Update `tasks/lessons.md` after corrections

## Core Principles

- **Simplicity First**: Make every change as simple as possible. Impact minimal code.
- **No Laziness**: Find root causes. No temporary fixes. Senior developer standards.
- **Minimal Impact**: Changes should only touch what's necessary. Avoid introducing bugs.

09.5 — The Claude Configuration Cascade

How Teams Structure .claude/ Across Org, Project & Developer Layers

CLAUDE.md and .claude/ aren't just one file in one folder — they cascade. Org-wide rules, project-specific context, and personal preferences compose automatically as Claude walks up the directory tree. Here's how to architect the layers so team-shared rules stay in git, personal preferences stay local, and nothing gets duplicated.

The .claude/ directory (project layer)

.claude/ at project root · committed to git · shared with team

The project-level .claude/ folder is the control center for Claude Code. It defines instructions, custom commands, permission rules, and specialized agent behaviors for everyone working in the repo.

CLAUDE.md

Central instruction manual. Claude reads it at session start for project context, architecture, and coding standards.

settings.json

Permission controls — which tools (bash, read, write) are allowed, which files or commands are denied, which MCP servers are available.

rules/

Modular, markdown-based instructions. Rules without a paths field apply globally; others only when working in designated directories.

commands/

Custom slash commands (e.g. /project:review). Each markdown file becomes a command, with shell support via the ! prefix.

skills/

Reusable, context-aware workflows. More advanced than commands — auto-invoked by Claude when relevant, not just triggered manually.

agents/

Specialized subagents with defined personas and restricted tool access. Run risky tasks (reviews, audits) in isolated sessions.

The ~/.claude/ directory (user layer)

~/.claude/ · personal to each developer · never committed to git

The global folder holds personal preferences that apply across every project on your machine. It's yours alone — never mandate its contents across a team.

~/.claude/CLAUDE.md

Global coding preferences — "always use TypeScript", "prefer functional style", personal workflow quirks.

~/.claude/projects/

Auto-memory store. Claude persists MEMORY.md per project so knowledge survives across sessions.

commands/ & skills/

Personal commands and skills available everywhere, not scoped to one project. Your personal toolbox.

Org and project .claude/ live in git. Only ~/.claude/ stays local. The config-cascade rule

Cascading rules & precedence

CLAUDE.md files are loaded by walking up the directory tree from the current working directory. Multiple CLAUDE.md files combine — each layer adds context without requiring duplication. When rules conflict, precedence flows:

System policy  ─▶  User (~/.claude/)  ─▶  Project (.claude/)   ← wins
   (floor)         (personal prefs)        (team-shared, highest)

Project-level configuration takes highest precedence, ensuring team-shared rules always apply. System policy sets the safety floor. User preferences sit in between.

Path-Scoped Rules

Rules inside .claude/rules/ can declare a paths: field to apply only when Claude works in specific subdirectories (e.g. src/api/). Rules without paths apply project-wide. Use this to keep CLAUDE.md lean instead of overloading it with conditionals.

8 Recommendations for Teams

1. CLAUDE.md is your contract

Org-level sets non-negotiable rules. Project-level extends with domain context. The directory tree walk combines them automatically — never duplicate.

2. Use rules/ for path scoping

Don't overload CLAUDE.md with conditional logic. Put API rules in rules/api.md scoped to src/api/, test rules in rules/tests.md. Modular and clean.

3. Skills are portable packages

Encode repeatable workflows as SKILL.md directories. Any methodology becomes a mountable package any project can adopt. Skills auto-invoke — commands don't.

4. Agents isolate risky tasks

Define subagents for sensitive operations — code reviews, security audits, dependency checks. They run in isolated sessions with restricted tool access.

5. Commands are the team API

Slash commands (/project:review, /project:deploy) make workflows discoverable. Org commands set the floor; projects add domain-specific ones.

6. Let MEMORY.md accumulate

Claude auto-persists session knowledge to ~/.claude/projects/*/MEMORY.md. Let it build context over time and review periodically for accuracy.

7. Version-control shared layers

Org and project .claude/ live in git. Only ~/.claude/ stays local. Skills and agents are versioned and distributed like code.

8. Scope MCP servers per layer

Org settings.json defines shared servers (Slack, Drive). Projects add domain-specific ones. Users add personal tools. Servers compose additively.

Notable settings.json Keys

Beyond permissions and MCP server lists, settings.json accepts several useful configuration keys:

  • prUrlTemplate — Point the footer PR badge to a custom review URL instead of github.com. Useful for GitLab, Bitbucket, or GitHub Enterprise teams.
  • otelHeadersHelper — Path to a script that auto-refreshes OTEL auth tokens (runs every 29 min). See Monitoring section.
  • disableAllHooks — Set true to disable all hooks globally (useful for troubleshooting).

Patterns & Anti-Patterns

✕ Copying CLAUDE.md into every repo

Write once at the org level and inherit via the directory tree walk. Duplication guarantees drift.

✕ Domain rules at the org level

Keep domain context at the project level. Org level is for cross-cutting standards only.

✕ One 2000-line CLAUDE.md

Factor into rules/ with path scoping for modularity. Bloated CLAUDE.md files lose important rules in the noise.

✕ Mandating personal ~/.claude/

Let developers own their personal layer entirely. Enforcement belongs in the project .claude/, not the user's home.

✕ Running sensitive tasks in the main session

Define isolated agents/ with restricted permissions for audits, deploys, and anything touching credentials.

✕ Ignoring MEMORY.md

Stale auto-memory poisons future sessions. Periodically review for accuracy and prune drift.

External Memory: Obsidian & Markdown Vaults

Claude's MEMORY.md is passive — session knowledge auto-persists as the model works. A curated markdown vault (Obsidian, Logseq, plain notes/) is the complement: intentional, human-authored, and portable across tools, models, and years.

Andrej Karpathy popularized this pattern — a long-lived Obsidian vault of notes, decisions, and research that gets selectively fed into LLM sessions as high-signal context. The vault outlives any single model or IDE; the AI is just one of many readers.

Practical integration patterns:

  • Symlink a vault subfolder into the repo (e.g. docs/notes/ → ~/Obsidian/project-x/) so Claude reads it as part of the project tree.
  • Paste relevant notes into context at session start — curated beats comprehensive.
  • Wire an Obsidian / Reflect / Notion MCP server so Claude can search and append notes directly, turning the vault into a read/write memory layer.
  • Treat the vault as source-of-truth; let MEMORY.md churn freely underneath it.
Codex Equivalent

OpenAI's Codex CLI uses the same layered architecture with different filenames: AGENTS.md instead of CLAUDE.md, .rules/ for guardrails, and .agents/skills/ for reusable workflows. The cascade model (org → project → user) is identical. See the Codex Best Practices page for the full mapping.

Setting Up MCP Connections via /plugin

The fastest way to connect Claude Code to external tools is through the /plugin command. Plugins bundle MCP servers with OAuth — no API tokens or manual config needed.

1

Open the Plugin Browser

In any Claude Code session, type /plugin and press Enter. This opens the interactive plugin browser showing all available plugins.

3

Authenticate via OAuth

Specialist Definitions

Each card mirrors the live specialist file under .claude/skills/sec-review-team/specialists/. baseline specialists always run; stack-specific specialists join based on stack-signals.md.

secrets-crypto-auditor baseline

▼

Primary scope

Secrets and cryptography.

  • Hardcoded secrets / API keys / tokens / passwords in source (full-tree grep on password|secret|api_key|token|bearer|credentials|private_key).
  • Weak hash algorithms (MD5, SHA1, DES, RC4).
  • Insecure randomness (Math.random() for security IDs; OS-level CSPRNG required for keys).
  • Improper key/token storage (plaintext on disk, unencrypted SQLite, localStorage for secrets).
  • Credentials in logs, error messages, panics, stack traces.
  • unsafe Rust touching key material.
  • TLS/SSL configuration — outdated protocol versions, weak cipher suites, certificate pinning where needed.
  • Key rotation / expiry policy.

Threat model

Frame against the realistic adversary for the stack: disk compromise (stolen laptop, forensics, backup snooping) for local-only desktop; network attacker plus insider for SaaS backends; secrets leaked in build logs and committed .env for the supply chain.

Coverage categories

hardcoded-secrets, weak-crypto, insecure-random, plaintext-key-storage, credential-logging, tls-config, key-rotation

dependency-supplychain-auditor baseline

▼

Primary scope

Supply chain — dependencies, lockfile integrity, install scripts, build trust.

  • npm: package.json, package-lock.json, npm audit if available; pinned vs floating versions; postinstall / install scripts.
  • Cargo: Cargo.toml, Cargo.lock, cargo audit.
  • Python: requirements.txt, Pipfile.lock, poetry.lock, pip-audit.
  • OSV: osv-scanner across any ecosystem.
  • Suspicious packages, typosquats, recently compromised deps.
  • build.rs, postinstall/preinstall scripts, Dockerfile RUN installs.
  • Plugin / core version alignment.

Threat model

Treats package manifests, lockfiles, and install scripts as a third-party trust boundary — known CVEs, typosquats, and post-install hooks compromise the build before any of your code runs. Will not invent CVE IDs if a scanner isn't available; reports tool-availability in coverage.

Coverage categories

known-cves, lockfile-integrity, version-pinning, install-scripts, typosquats, plugin-alignment, build-rs-trust

silent-failure-hunter baseline

▼

Primary scope

Error paths that fail open or mask security-relevant failures.

  • Swallowed exceptions: try{}catch{}, .catch(() => {}), Rust .ok() / .unwrap_or_default() on security-relevant errors.
  • Fallback paths that bypass validation when the primary check errors.
  • Rust match arms with Err(_) => Ok(()).
  • Async promises without .catch; unhandled rejections.
  • unwrap() on user-influenced data.
  • React error boundaries catching but not surfacing state-corrupting errors.
  • Errors around DB migrations, capability checks, session-rotation — not the check itself, the handling of the failure.

Threat model

Security-relevant silences only. A swallowed UI toast error is not a finding; a swallowed DB-migration error that leaves the schema half-applied is. Defers architectural IPC/capability issues to auth-authz-auditor and missing validation to input-validation-auditor.

Coverage categories

swallowed-exceptions, fallback-bypasses-validation, unhandled-rejections, unwrap-on-user-input, error-boundary-absorption, migration-silent-failures

data-exposure-auditor baseline

▼

Primary scope

Unintended exposure of sensitive data.

  • PII / sensitive content storage (DB path, encryption-at-rest, cache/temp files, exports).
  • Log redaction (println!, console.log, log::*, tracing::*).
  • Error-message shapes leaking internals (stack traces, file paths, SQL).
  • IDOR / tenant isolation.
  • Clipboard / drag-and-drop lingering data.
  • Export pipeline (PDF/DOCX temp files, cleanup).
  • Outbound network calls — if threat model claims "no network", verify with grep (any hit is a finding).

Threat model

Verifies stated threat-model claims (e.g. "no network") with explicit grep — any deviation is a finding. Owns "what could leak" (storage, logs, errors); defers analytics/consent to privacy-telemetry-auditor and CSP/CORS to frontend-security-auditor when those are in roster, and reclaims them otherwise.

Coverage categories

pii-storage, log-redaction, error-leak, idor-tenant-isolation, clipboard-dnd, export-pipeline, outbound-calls-claimed-none, (conditionally) csp-cors-sri-cookies

auth-authz-auditor stack-specific

▼

Primary scope

Authentication, authorization, session management, and privilege boundaries.

  • Auth flows (login, signup, logout, password reset, MFA, SSO).
  • Session handling (cookie lifetime, rotation on privilege change, invalidation on logout).
  • JWT / OAuth / SAML — signature verification, audience/issuer checks, token expiry, refresh flows.
  • RBAC / ABAC — role definitions, assignment, enforcement at every protected operation.
  • Privilege boundaries — frontend/webview vs backend/trusted side, Tauri IPC command surface, Electron preload scripts, microservice trust.
  • Capability grants (Tauri capabilities/*.json, browser CSP script-src, OS permissions).
  • Endpoint / IPC command authorization mapping: every protected surface ↔ its required capability.

Threat model

Maps every endpoint, IPC command, or authenticated surface to its required capability. For local-only or single-user apps, the focus shifts from user-level auth to IPC boundaries and capability grants between webview and native side. Architectural IPC/capability issues are this specialist's scope; error-path silences belong to silent-failure-hunter.

Coverage categories

auth-flows, session-handling, jwt-oauth-saml, rbac-abac, privilege-boundaries, ipc-capability-grants

input-validation-auditor stack-specific

▼

Primary scope

Injection vectors — tracing every external input to its sink.

  • SQL injection — string-interpolated SQL, template literals, FTS5 MATCH syntax (parameterization alone doesn't save you from FTS5 query-syntax injection).
  • Command / path traversal — fs::*, Command::*, exec, os.system, dialog APIs, archive extraction (zip-slip).
  • XSS — React's unsafe HTML escape hatch (dangerouslySetInnerHTML), innerHTML/outerHTML, Markdown→HTML serializers, template engines with auto-escape disabled.
  • Deserialization — JSON/YAML/bincode/pickle boundaries; prototype pollution.
  • IPC payload validation — backends validating incoming args via typed structs vs trusting the caller.
  • Open-redirect / SSRF — URL parameters reaching fetch/reqwest/curl without host allowlist.

Threat model

Trace every external input to its sink. Defers prompt injection to prompt-injection-auditor when in roster; otherwise reclaims it. Authentication of input sources belongs to auth-authz-auditor; validation of input content is here.

Coverage categories

sql-injection, fts5-syntax-injection, command-injection, path-traversal, zip-slip, xss, deserialization, ipc-payload-validation, ssrf, open-redirect

frontend-security-auditor stack-specific

▼

Primary scope

Frontend / web-platform security.

  • CSP — header presence, strictness; absence of unsafe-inline, unsafe-eval, remote origins without nonce/hash.
  • CORS — Access-Control-Allow-Origin: * on credentialed endpoints, missing Vary: Origin.
  • Cookie flags — Secure, HttpOnly, SameSite=Strict/Lax on session cookies.
  • Subresource Integrity (SRI) — on <script src=> and <link> to third-party CDNs.
  • postMessage / BroadcastChannel — origin verification.
  • Clickjacking — X-Frame-Options / frame-ancestors CSP directive.
  • Third-party script hygiene — analytics, fonts, CDN scripts; async loading patterns.

Threat model

Owns the web-platform attack surface — CSP, CORS, SRI, cookie flags, postMessage, clickjacking — that data-exposure-auditor defers when this specialist is in the roster. Cross-cuts with privacy-telemetry-auditor on third-party script inclusion.

Coverage categories

csp, cors, cookie-flags, sri, postmessage-origin, clickjacking, third-party-scripts

prompt-injection-auditor stack-specific

▼

Primary scope

LLM / agentic security.

  • Untrusted input reaching tool-use arguments (direct or via system-prompt concat).
  • Prompt assembly via string concat — f"You are {role}. User: {msg}" patterns.
  • Missing output sanitization — LLM response rendered as HTML, executed as code, or used in SQL unvalidated.
  • Tool-use role confusion — tools giving the model power the user shouldn't have without auth gates.
  • Context poisoning — untrusted RAG ingests without provenance tracking.
  • Memory / skill / system-prompt injection — user content altering agent memory or next-turn system prompts.
  • Indirect prompt injection via external data (web pages, emails, PDFs, Slack) fetched into context.
  • Secret exfiltration via prompts — LLM instructed to echo env vars / config / other users' data.

Threat model

Trace every untrusted input to prompt-assembly sinks. Owns prompt injection when in roster; input-validation-auditor defers. Cross-cuts with data-exposure-auditor (LLM exfil) and secrets-crypto-auditor (leaked keys via prompt-echo).

Coverage categories

prompt-concat-untrusted, tool-use-role-confusion, output-sanitization, context-poisoning-rag, memory-skill-injection, indirect-injection, secret-exfil-via-prompt

iac-auditor stack-specific

▼

Primary scope

Infrastructure-as-code review.

  • Public-by-default resources (S3 buckets, RDS, blob storage) — ACL, bucket policy, encryption-at-rest.
  • Wildcards in IAM / RBAC (Action: "*", Resource: "*", cluster-admin bindings).
  • Missing encryption (TLS in transit, KMS at rest, SSM parameters marked SecureString).
  • Network policies — overly broad 0.0.0.0/0 ingress, missing egress restrictions.
  • Kubernetes pod security — runAsRoot, privileged, allowPrivilegeEscalation, missing readOnlyRootFilesystem, hostPID/hostNetwork/hostIPC.
  • Tagging / naming for audit trail.
  • Hardcoded credentials in *.tfvars, values.yaml, .env.tpl.

Threat model

Reviews Terraform / CloudFormation / Helm / Kustomize / plain-YAML K8s manifests for cluster-level resources and IAM correctness. Defers credentials-in-plain-config to secrets-crypto-auditor via cross-reference; coordinates pod-level security context with container-runtime-auditor.

Coverage categories

public-resources, iam-wildcards, encryption-at-rest, encryption-in-transit, network-policies, pod-security-standards, iac-hardcoded-credentials, tagging-audit-trail

container-runtime-auditor stack-specific

▼

Primary scope

Container and runtime security.

  • Dockerfile — FROM pinning (digest, not just tag), USER not root, COPY not ADD for local, --chown, no secrets in layers.
  • Base image freshness and known CVEs.
  • RUN shell-injection risk via build-args.
  • docker-compose.yml — privileged: true, exposed ports, volume mounts into /etc, /var/run/docker.sock.
  • Kubernetes pod spec — runAsRoot, allowPrivilegeEscalation, capabilities.add, missing readOnlyRootFilesystem, hostPID/hostNetwork/hostIPC.
  • Helm chart defaults that open ports or privileges.

Threat model

Owns Dockerfile, compose, pod spec, and Helm values. Coordinates with iac-auditor (cluster-level resources) and dependency-supplychain-auditor (base-image CVEs).

Coverage categories

dockerfile-hardening, base-image-cves, compose-privileges, pod-security-context, helm-default-exposure

ci-cd-security-auditor stack-specific

▼

Primary scope

CI/CD pipeline security.

  • GitHub Actions: pull_request_target on workflows that check out untrusted PR code (classic RCE pattern).
  • Third-party actions not SHA-pinned — prefer @<sha> over @v4.
  • permissions: block — default read-all; explicit escalation only where needed.
  • Secrets exposure — echo ${{ secrets.X }} in logs, env: pass-through to untrusted steps.
  • on: pull_request + secrets reachable from forks.
  • GitLab CI / CircleCI / Jenkins equivalents.
  • Artifact integrity — signed builds, SLSA provenance, cosign attestations.
  • Build-time dep fetch — curl | sh installs, unpinned deps in scripts.

Threat model

CI/CD pipeline trust boundaries — workflow triggers, action SHA-pinning, secrets exposure in logs and forks, permissions hardening, artifact integrity. Cross-cuts with dependency-supplychain-auditor (action-version supply chain) and secrets-crypto-auditor (leaked secrets in CI logs).

Coverage categories

workflow-triggers, action-pinning, permissions-block, secrets-in-ci, fork-secret-access, artifact-integrity, build-time-dep-fetch

concurrency-race-auditor stack-specific

▼

Primary scope

Race conditions and concurrency hazards with security impact.

  • TOCTOU: check-then-act without atomicity (if os.path.exists(path): open(path); auth read-then-act without transaction).
  • Double-check-locking anti-patterns.
  • Shared state without synchronization — globals mutated from async handlers, React state writes from overlapping effects.
  • Rate-limit bypass via concurrency — non-atomic counter read+increment.
  • Session fixation / replay — session IDs accepted before validation; concurrent login creating multiple session rows.
  • Transactional integrity — DB writes that should be atomic split across statements; UPDATE balance without row-level locking.
  • Rust Send/Sync misuse — unsafe impl Send/Sync, Arc<Mutex<_>> escape.
  • Async cancellation safety — futures leaving state half-modified if cancelled.

Threat model

Concurrency correctness in security-relevant code paths — money, permissions, sessions, rate limits. Cross-cuts with silent-failure-hunter (swallowed errors in race paths) and auth-authz-auditor (session races) via cross-reference.

Coverage categories

toctou, double-check-locking, shared-state-unsynced, rate-limit-race, session-race, transactional-integrity, async-cancellation

privacy-telemetry-auditor stack-specific

▼

Primary scope

What the app phones home about, what data leaves, regulatory (GDPR/CCPA) signals.

  • Outbound data inventory — every fetch/XMLHttpRequest/reqwest/axios destination + payload. Does any include PII?
  • Analytics / tracking SDKs — GA, Segment, Mixpanel, PostHog, Sentry, FullStory. Opt-in vs opt-out. DNT / GPC respected. Load before consent gathered?
  • Cookie inventory — domain, expiry, flags, purpose; categorized as strictly-necessary / functional / analytics / marketing.
  • Storage inventory — localStorage, sessionStorage, IndexedDB, cache APIs; client-side PII with retention.
  • Third-party script / iframe inclusion — what domains load at page start; SRI usage.
  • Cross-border data transfer — servers / CDNs / logging endpoints in unexpected jurisdictions.
  • Consent flow — banner presence, actual gating of analytics/marketing SDKs (classic bug: banner shows but scripts load anyway).
  • Retention / deletion — DSAR flow, logs retention visible.
  • Desktop telemetry — Sentry, crash reporters, auto-update pings, product analytics.

Threat model

Owns "what deliberately leaves + regulatory"; data-exposure-auditor owns "what could leak". Catches the classic consent-banner bug where SDKs load before the user clicks accept.

Coverage categories

outbound-destinations, analytics-sdks, cookies, client-storage-pii, third-party-scripts, cross-border-transfer, consent-flow-gating, retention-dsar

# Multi-Agent Security Review Team — Orchestration

Follow these steps. Review-only. Never apply fixes.

## Step 0: Pre-run confirmation

The skill's largest user-facing safety gate. Never spawn specialists without showing this block (unless `--yes` / `auto-approve` is passed).

### 0.1 Parse arguments

- **Target path** — absolute path. If missing, ask: *"What repo should I review? (absolute path)"* Verify it exists and is readable.
- **Scope** — one of:
  - `all` — full tree
  - `<subdir>` — a subdirectory, e.g. `src/api/`
  - `diff vs main` (or `diff vs <branch>`) — only files changed on current branch vs target branch
  If missing, ask: *"What scope? Full tree, a subdirectory, or a diff?"*
- **Flags:** `--yes` / `-y` skips confirmation entirely. Default is to confirm.

### 0.2 Enumerate scope

Use git-aware commands (fall back to `find` if not a git repo):

```bash
# File count + line count (scoped)
cd "<TARGET>"
if [[ "<SCOPE>" == "all" ]]; then
  FILE_COUNT=$(git ls-files | wc -l)
  LINE_COUNT=$(git ls-files | xargs wc -l 2>/dev/null | tail -1 | awk '{print $1}')
elif [[ "<SCOPE>" == diff* ]]; then
  BASE="${SCOPE#diff vs }"; BASE="${BASE:-main}"
  FILES=$(git diff --name-only "$BASE"...HEAD)
  FILE_COUNT=$(echo "$FILES" | wc -l)
  LINE_COUNT=$(echo "$FILES" | xargs wc -l 2>/dev/null | tail -1 | awk '{print $1}')
else
  FILE_COUNT=$(git ls-files "<SCOPE>" | wc -l)
  LINE_COUNT=$(git ls-files "<SCOPE>" | xargs wc -l 2>/dev/null | tail -1 | awk '{print $1}')
fi
```

Cap `LINE_COUNT` at 500k (anything larger: warn and suggest scope narrowing).

### 0.3 Detect stack signals

Read the highest-signal manifests first (cheapest): `README.md` top 50 lines, `package.json`, `Cargo.toml`, `pyproject.toml`, `go.mod`, `Gemfile`, `tauri.conf.json`, `Dockerfile`, `.github/workflows/` (existence only), `requirements.txt`. Two or three file-reads is enough.

Apply each rule in `stack-signals.md` against what you observe. Collect matching signals (multiple can match). Example: a Tauri note app emits `desktop` + `desktop-with-webview`.

### 0.4 Pick default roster

Per `stack-signals.md`: union the specialist sets across all matching stack signals. Always include the baseline 4 (`secrets-crypto-auditor`, `dependency-supplychain-auditor`, `silent-failure-hunter`, `data-exposure-auditor`).

### 0.5 Detect scanner availability

For each scanner in Step 2.5's pre-pass list, run `command -v <name>`. Flag available vs missing. Missing scanners don't block — they just reduce pre-pass coverage; dependent specialists will work in manual mode.

### 0.6 Estimate runtime + tokens + cost

Use these baselines (from CBP-060 reference run against `recall` on Opus 4.7):

- **Input tokens per specialist:** `~min(40k, 30 + 0.02 × LINE_COUNT)` k tokens — small repos float near 30k overhead; large repos grow linearly with source lines but the agent self-caps reads.
- **Output tokens per specialist:** `~5–10k` regardless of repo size (findings + prose).
- **Total tokens:** `N_specialists × (input + output)`.
- **Runtime:** longest-path specialist `≈ 3 min` baseline; add `0.6 sec / file read`. Dependency auditor is typically slowest (`~20–30 min` on a repo with a 700-package `node_modules`). Report estimate as a range: `longest × 1.1` to `longest × 1.5`.
- **Cost** (on Opus 4.7 pricing — $5/MTok input, $25/MTok output, April 2026): `(input_tokens × 5 + output_tokens × 25) / 1_000_000`. Print `$X.XX`.

Use `claude-opus-4-7[1m]` pricing if the orchestrator is running on the 1M-context variant; otherwise standard Opus 4.7. Print the pricing assumption.

### 0.8 Handle response

- **y** / **yes** / (`--yes` flag) → continue to Step 1.
- **narrow** → prompt: *"Narrow to which subdirectory or diff?"* — re-run 0.2–0.7 with the new scope.
- **add-specialist `<name>`** → append the named specialist if present in `specialists/`, reject if unknown, re-render 0.7.
- **drop-specialist `<name>`** → remove from roster, warn if dropping a baseline specialist (*"secrets-crypto is in the baseline. Drop anyway?"*), re-render 0.7.
- **abort** / **n** / **no** / empty response after 60 s (if interactive) → exit cleanly. Do NOT create `.planning/security-review/`. Do NOT spawn any agent.

## Step 1: Final roster

Load only the specialist brief files from `specialists/` that match the confirmed roster. Do not load briefs for specialists not in the roster.

## Step 2: Prepare findings directory

Create `<TARGET>/.planning/security-review/` if absent.

**Pollution check:** run `git check-ignore <TARGET>/.planning/` (or grep `.gitignore`). If `.planning/` is NOT ignored, warn the user:
> "⚠ `.planning/` is not in the target's .gitignore. Findings land inside the repo. Options: (a) add to .gitignore, (b) write to /tmp/sec-review-<timestamp>/ instead, (c) proceed."

Respect choice. Default (no response) is (c).

## Step 2.5: Scanner pre-pass

Deterministic scanners run before the LLM specialists. This is the biggest accuracy-per-token lever in the skill — specialists spend tokens on signal tools can't catch (architectural, multi-step, context-dependent), not re-deriving secret-detection regexes.

### 2.5.1 Scanner inventory

| Scanner | Detects | Command | Feeds specialist |
|---|---|---|---|
| `gitleaks` | secrets at rest | `gitleaks detect --source <TARGET> --report-format json --report-path -` | secrets-crypto-auditor |
| `trufflehog` | secrets (git history + fs) | `trufflehog filesystem <TARGET> --json` | secrets-crypto-auditor |
| `semgrep` | pattern-based SAST | `semgrep scan --config auto --json --output -` | multi: input-validation, data-exposure, silent-failure (by rule category) |
| `npm audit` | npm CVEs | `cd <TARGET> && npm audit --json` | dependency-supplychain-auditor |
| `cargo audit` | crate CVEs | `cd <TARGET> && cargo audit --json` | dependency-supplychain-auditor |
| `pip-audit` | Python CVEs | `pip-audit -r <TARGET>/requirements.txt -f json` | dependency-supplychain-auditor |
| `osv-scanner` | multi-ecosystem CVEs | `osv-scanner -r <TARGET> --format json` | dependency-supplychain-auditor |
| `bandit` | Python SAST | `bandit -r <TARGET> -f json` | input-validation-auditor (Python targets) |
| `hadolint` | Dockerfile lint | `hadolint <file> -f json` | container-runtime-auditor |
| `trivy image` | base image CVEs | `trivy image --format json <image>` | container-runtime-auditor |
| `actionlint` | GHA syntax + security | `actionlint -format '{{json .}}'` | ci-cd-security-auditor |
| `tfsec` | Terraform security | `tfsec <TARGET> --format json` | iac-auditor |
| `checkov` | IaC security | `checkov -d <TARGET> --output json` | iac-auditor |
| `retire.js` | known-vulnerable JS libs | `retire --path <TARGET> --outputformat json` | frontend-security-auditor |

Full mapping: see [`docs/scanner-coverage.md`](docs/scanner-coverage.md).

### 2.5.2 Detection + invocation

For each scanner in the inventory:

```bash
if command -v <scanner> >/dev/null 2>&1; then
  run the scanner, capture stdout
  parse JSON output
  normalize to finding.schema.json format (see 2.5.3)
  append to <TARGET>/.planning/security-review/scanner-prepass.jsonl
else
  log "scanner <name> not available — skipping"
fi
```

Run scanners in parallel where safe (no shared-state conflicts). Wall-clock ceiling per scanner: 120 s (kill if exceeded; log partial results).

### 2.5.3 Normalization

Every scanner emits different JSON. Normalize each finding to `finding.schema.json` with:

- `id` — `scanner-<name>-<index>` (e.g. `scanner-gitleaks-001`)
- `specialist` — the specialist this finding is routed to (see inventory above). Even scanner findings get routed; specialists triage them.
- `source` — `"scanner-<name>"` (so provenance is visible)
- `severity` — mapped from scanner's severity (`HIGH` → `high`, `MEDIUM` → `medium`, etc.)
- `confidence` — scanners get `certain` for deterministic rule hits, `likely` for heuristic rules
- `title`, `root_issue` — from scanner output, slug-normalized
- `file`, `line_range`, `evidence` — from scanner output
- `exploit`, `fix` — scanners often omit these; leave `""` and let the specialist agent fill during triage

Save to `<TARGET>/.planning/security-review/scanner-prepass.jsonl`.

### 2.5.4 Feed specialists

When spawning each specialist in Step 3, include a "Pre-pass context" section in the brief:

```
Pre-pass scanner findings relevant to your specialist (from `<TARGET>/.planning/security-review/scanner-prepass.jsonl`):

  [N findings from gitleaks]
  [M findings from semgrep for your category]

Triage rules:
  1. For each scanner-sourced finding, decide: true-positive / false-positive / needs-investigation.
     Emit a finding entry in your JSONL with `source: "scanner-<name>-confirmed"` (for true-positive) or
     `source: "scanner-<name>-false-positive"` (for false-positive triaged out).
  2. After triage, find what the scanner missed via architectural / cross-file analysis — these go in your
     JSONL with `source: "specialist"` as usual.
```

Specialists are not obligated to re-derive secret regexes. Their job is triage + judgment.

### 2.5.5 REPORT.md provenance

The consolidation step (Step 4) renders `REPORT.md` with a "Source" column for each finding so the reader sees which scanner (if any) caught it and whether a specialist confirmed or overturned the triage. Scanner-only confirmations (no specialist review) are flagged with `⚠ unreviewed by specialist` — this happens when a specialist errored before triaging the scanner pre-pass.

### 2.5.6 Scanner unavailability

If zero scanners are available in the environment, the run continues — specialists do the full job themselves, which is the CBP-060 baseline mode. Log a tooling caveat in REPORT.md so the reader knows coverage is LLM-only.

## Step 3: Spawn specialists in parallel

Issue one `Agent` tool call per specialist in a **single assistant message** — this is what makes them run concurrently.

For each specialist:

- Read its brief file (`specialists/<name>.md`).
- Substitute `<TARGET>`, `<SCOPE>`, `<STACK CONTEXT>`, `<THREAT_MODEL>`.
- Pass `allowed-tools` per the specialist's `## Allowed tools` section (hard read-only; Write scoped to the four output-file paths only).
- Use `preferred_subagent_type` from frontmatter; fall back to `fallback_subagent_type` if unregistered.

Print to the user before the parallel batch returns:

```
Spawning <N> specialists in parallel against <TARGET> (scope: <SCOPE>).

Each specialist writes to <TARGET>/.planning/security-review/<name>.{md, findings.jsonl, coverage.jsonl, status.json}.

Live progress (run in a separate terminal):
  watch -n 5 'jq -s "sort_by(.agent) | map({agent, status, files_read, findings_written})" <TARGET>/.planning/security-review/*.status.json'

Or:
  tail -f <TARGET>/.planning/security-review/*.status.json

Spawning now. Interim status lines will appear as each specialist returns.
```

### 3.1 Heartbeat contract (per specialist)

Every specialist brief instructs the agent to write `<name>.status.json` at spawn time and to update it as progress happens. Format:

```json
{
  "agent": "<specialist-name>",
  "status": "starting | scanning | writing-findings | completed | errored",
  "started_at": "<ISO>",
  "finished_at": "<ISO|null>",
  "files_read": <int>,
  "findings_written": <int>,
  "current_file": "<path|null>",
  "severity_counts": { "critical": 0, "high": 0, "medium": 0, "low": 0 },
  "error": "<message|null>"
}
```

- On spawn: write with `status: "starting"`, `started_at`, counters at 0.
- Every ~5 file reads: update `files_read`, `current_file`, set `status: "scanning"`.
- On each finding: increment `findings_written`, update `severity_counts`.
- On completion: set `status: "completed"`, `finished_at`, final counts.
- On error: set `status: "errored"`, `finished_at`, populate `error`.

The agent must flush `.status.json` to disk **before returning from its `Agent` tool call**, so the orchestrator sees the final state once the subagent completes.

### 3.2 Interim reports (orchestrator-side)

Batched `Agent` calls in a single message run concurrently, but the orchestrator receives their results serialized — one completion at a time. As each specialist returns:

1. Read the specialist's `status.json` file to confirm final state.
2. Print **one line** to the user:

```
✓ auth-authz-auditor       done    (0C / 2H / 2M / 1L)   [4m 22s]   12 files read
✓ input-validation-auditor done    (0C / 0H / 0M / 3L)   [3m 15s]    8 files read
✗ dependency-supplychain-auditor errored after 18m — node_modules walk timed out. Partial findings preserved.
✓ silent-failure-hunter    done    (0C / 2H / 3M / 3L)   [5m 03s]   14 files read
...
```

3. Errored specialists don't block the run. Keep processing completions from the remaining specialists. In Step 4 (consolidate), errored specialists are flagged in REPORT.md with their last-known `status.json` state.

4. After the last specialist returns, print:
```
All <N> specialists completed (<X> done, <Y> errored). Consolidating...
```

### 3.3 Output files per specialist

Each specialist writes four files:

- `<name>.md` — prose findings (human-readable)
- `<name>.findings.jsonl` — per `schema/finding.schema.json`
- `<name>.coverage.jsonl` — per `schema/coverage.schema.json`
- `<name>.status.json` — heartbeat / final state

## Step 4: Consolidate

Read all `*.findings.jsonl` and `*.coverage.jsonl`. Apply `docs/consolidation-template.md`:

1. Group findings by `root_issue` (dedupe).
2. For each group: `max(severity)`, `max(confidence)`, `distinct(specialist) → raised_by`.
3. Split actionable (`certain|likely`) from worth-investigating (`possible|unverified`).
4. Build coverage matrix (category × specialist) from coverage.jsonl.
5. Render `REPORT.md` from template.

If any specialist failed schema validation on its JSONL, fall back to prose parse for that specialist and log a warning in REPORT.md.

As each specialist returns, print an interim-status line: `✓ <name> done (<counts>) [<runtime>]` or `✗ <name> errored: <msg>`. Don't block on errored specialists.

## Step 5: Deliver

Tell the user:
- Path to `REPORT.md` and the four per-specialist files each
- Headline severity counts (deduped)
- Top 2–3 fixes by impact
- Tooling caveats (scanners unavailable, fallbacks taken, specialists errored)
- Coverage completeness score

Do NOT apply fixes. Fix workflow → companion skill `/sec-review-fixes` (CBP-070).

# secrets-crypto-auditor

## Primary scope
Secrets and cryptography.

- Hardcoded secrets / API keys / tokens / passwords in source (full-tree grep: `password|secret|api[_-]?key|token|bearer|authorization|credentials|private[_-]?key`).
- Weak hash algorithms (MD5, SHA1, DES, RC4).
- Insecure randomness (`Math.random()` for security IDs, `rand::thread_rng()` where CSPRNG needed; OS-level CSPRNG required for keys).
- Improper key/token storage (plaintext on disk, unencrypted SQLite, localStorage for secrets).
- Credentials in logs, error messages, panics, stack traces.
- `unsafe` Rust touching key material.
- TLS/SSL configuration — outdated protocol versions, weak cipher suites, certificate pinning or its absence where needed.
- Key rotation / expiry policy.

## Threat-model framing
Frame findings against the realistic adversary for the stack:
- **Local-only desktop:** disk compromise (stolen laptop, forensics, backup snooping) — plaintext DB is the top risk.
- **SaaS backend:** network attacker + insider — key storage in secret manager, rotation, scoped IAM.
- **Supply-chain:** secrets leaked in build logs, committed `.env`, compromised CI secrets.

## Overlap with other specialists
- **Primary owner of:** secrets in code, crypto primitive choice, key storage, randomness.
- **Cross-cuts with:**
  - `dependency-supplychain-auditor` — secrets leaked in CI workflows overlap; coordinate.
  - `privacy-telemetry-auditor` — data-handling compliance overlaps with key-handling; yours is the crypto/key side.

## Brief

> Review secrets and cryptography in `<TARGET>` (scope: `<SCOPE>`). Stack: `<STACK CONTEXT>`. Realistic threat model: `<THREAT_MODEL>`. Check: hardcoded secrets, weak crypto, insecure random, improper key storage, credential logging, `unsafe` Rust touching keys, TLS config, key rotation. Frame findings against the actual threat model.
>
> **Output contract (four files):** `secrets-crypto-auditor.{md, findings.jsonl, coverage.jsonl, status.json}`. Same format as other specialists — see `auth-authz-auditor.md` for details.
>
> **Confidence on every finding.** Negative findings require searches + search_limits in coverage.jsonl (e.g., "grepped for X|Y|Z, found nothing — would miss credentials assembled at runtime from env + obfuscation").
>
> **Hard rules:** read-only; cite evidence; no invented CVEs.
>
> Report: four output paths + severity count.

## Output files
`secrets-crypto-auditor.{md, findings.jsonl, coverage.jsonl, status.json}`

## Allowed tools

# REPORT.md Consolidation Template

This template defines how the orchestrator synthesizes `<agent>.findings.jsonl` and `<agent>.coverage.jsonl` from all specialists into a single deduped `REPORT.md`.

## Pipeline (pseudocode)

```
# 1. Read all findings
findings = []
for f in .planning/security-review/*.findings.jsonl:
    for line in f:
        finding = validate_against(finding.schema.json, parse(line))
        findings.append(finding)

# 2. Read coverage
coverage = []
for f in .planning/security-review/*.coverage.jsonl:
    for line in f:
        coverage.append(parse(line))

# 3. Dedupe by root_issue
groups = group_by(findings, key=root_issue)
for group in groups:
    merged = {
        root_issue: group[0].root_issue,
        severity: max_severity(group),  # critical > high > medium > low > info
        confidence: max_confidence(group),  # certain > likely > possible > unverified
        title: group[0].title,
        raised_by: distinct([f.specialist for f in group]),
        files: distinct([f.file for f in group]),
        exploits: [f.exploit for f in group],
        fixes: [f.fix for f in group],
        evidence: [f.evidence for f in group],
        count: len(group)
    }

# 4. Split by actionable vs worth-investigating
actionable = [g for g in merged_groups if g.confidence in (certain, likely)]
worth_investigating = [g for g in merged_groups if g.confidence in (possible, unverified)]

# 5. Coverage matrix
matrix = pivot(coverage, rows=category, cols=specialist, values=status+confidence)

# 6. Completeness score
score = (checked-clean + checked-issues-found) / total_categories
```

## Severity / confidence precedence

```
SEVERITY: critical > high > medium > low > info
CONFIDENCE: certain > likely > possible > unverified
```

When merging a group, `severity` takes the max (most severe wins), `confidence` takes the max (most certain wins). Rationale: if one specialist says `high/certain` and another says `medium/possible`, the issue is high severity with certainty — don't down-rank for disagreement.

## Dedupe rules

1. Findings share a `root_issue` → merge into one entry.
2. Findings share `file:line_range` but different `root_issue` → keep separate (they're different issues at the same location).
3. Findings with different `root_issue` but identical `title` or `fix` → orchestrator flags as "possibly related" but does not auto-merge (avoid collapsing distinct issues).

# Design Tradeoffs and Deliberate Non-Features

These are things that look like gaps but are deliberate choices. Captured here so future contributors don't re-raise them — or know exactly what would have to change to flip the decision.

## 1. `general-purpose` fallback is the intended path when typed plugin specialists aren't loaded

**What:** Each specialist brief lists a `preferred_subagent_type` (e.g., `security-auditor`, `pr-review-toolkit:code-reviewer`) and `fallback_subagent_type: general-purpose`. When the preferred type isn't registered, the orchestrator falls back without warning.

**Why:** Validated on Opus 4.7 in CBP-060 — quality of findings did not degrade when all 6 specialists ran as `general-purpose`. The brief is what produces focus, not the agent type. Forcing typed plugins would make the skill unusable in plain Claude Code installs.

**When to revisit:** If typed plugin specialists gain meaningful capability the brief alone can't replicate (e.g., pre-loaded scanner integration, specialized tool allowlists), make the fallback warn the user and offer to install the plugin.

---

## 2. Findings are written to `<TARGET>/.planning/security-review/` inside the target repo

**What:** Per-specialist files and the consolidated REPORT.md live inside the target's `.planning/` directory.

**Why:** Preserves history across runs (enables re-run mode, ledger), makes findings co-located with the code they describe, works with existing `.planning/` conventions in the FSAD Playbook and related projects. Moving to `/tmp/` loses all of this for very little privacy gain (the target-repo owner already has access).

**Mitigation:** Step 2 (`SKILL.md`) checks target `.gitignore` for `.planning/` and warns if absent, offering to write to `/tmp/sec-review-<timestamp>/` instead. This handles the minority of teams that don't gitignore `.planning/` and don't want internal findings committable.

**When to revisit:** If multiple users report accidental commits of findings — consider reversing the default.

---

## 3. No streaming / interim output during a specialist's run

**What:** The `Agent` tool is synchronous. The orchestrator blocks until a subagent returns. No incremental updates from a running specialist.

**Why:** Platform limitation, not a skill bug.

**Workaround (CBP-065):** Specialists write `<name>.status.json` files during their run. Users can `tail -f .planning/security-review/*.status.json` in a second terminal for live progress. As each agent returns, the orchestrator prints a one-line interim status.

**When to revisit:** If the `Agent` tool ever gains streaming semantics, remove the status-file dance and stream progress inline.

---

## 4. Two sources of truth: canonical prompt + embedded playbook copy

**What:** The original monolithic prompt lives at `markdown/design/Team_of_security_agents.md`. The playbook embeds a verbatim copy in the Security Review HTML section. They must be kept in sync manually.

**Why:** The playbook is a shareable static asset (one HTML file); the skill is an executable artifact. Both are legitimate distribution paths with different audiences.

**Mitigation (CBP-071):** Playbook refactors to render a summary + invocation example pointing at the canonical skill, rather than inlining the full prompt. Legacy prompt is preserved in a collapsible. This cuts most of the duplication.

**When to revisit:** If a future build pipeline can render the HTML section from the skill source directly, remove the duplicate.

---

## 5. No automatic fix application

**What:** The skill is explicitly review-only. It never edits files, never opens PRs, never applies fixes.

**Why:** Fix-application risk is very different from review risk — review reads code, fix writes code. Conflating them would break the predictability guarantee that makes the skill safe to run unattended.

**Complement (CBP-070):** `/sec-review-fixes` is a separate companion skill that consumes `REPORT.md` / `findings.jsonl` and proposes fix PRs with interactive approval gates. Separate skill = separate sandbox = separate approval surface.

**When to revisit:** Never. This separation is load-bearing.

---

## 6. "Review-only" is enforced by tool allowlists, not prompt instruction alone

**What:** Every specialist brief includes an `## Allowed tools` section. When the orchestrator spawns the agent, it passes `allowed-tools` restricting the agent to `Read`, `Grep`, `Glob`, a read-only `Bash` allowlist, and `Write` scoped to the four output file paths. `Edit`, unlisted `Bash`, and `Write` to anything else are **denied at the harness level**.

**Why:** Prompt-level "don't edit files" is a wish, not a guarantee. A misbehaving or prompt-injected agent could silently mutate the target. Tool allowlisting closes that gap with a hard enforcement boundary.

**When to revisit:** If a specialist provably needs a tool outside the current allowlist (e.g., a future `trivy apply` that only has read-only modes as flags), update the allowlist per-specialist rather than loosening globally.

---

## Summary: what this skill deliberately is NOT

- Not a replacement for human security review — a force-multiplier.
- Not a CI gate for new vulns — that's a scanner's job (the skill feeds scanners in via pre-pass and adds architectural signal on top).
- Not a fix-generating tool — that's `/sec-review-fixes`.
- Not a compliance audit — regulatory mapping is out of scope (see `privacy-telemetry-auditor` for telemetry/consent, which is as close as it gets).
- Not a pentest — no dynamic analysis, no exploitation, no runtime fuzzing.

If you want any of those, the skill is the wrong tool. These separations are intentional and load-bearing.

Skills are markdown directories that define specialized workflows. Claude loads them on demand — via slash commands or automatically by matching your task — providing consistency across sessions and team members.

What are Skills?

Skills are directories in .claude/skills/ containing a SKILL.md file plus optional supporting files (templates, scripts, reference docs). They turn tribal knowledge — code review checklists, deployment procedures, estimation frameworks — into repeatable, shareable workflows. Claude can auto-load skills by matching your task to skill descriptions, or you invoke them manually with /skill-name. Plugin skills use a namespace: /plugin:skill-name.

Skill File Anatomy

my-skill/
├── SKILL.md          # Required — skill content
├── reference.md      # Optional — detailed docs
├── examples.md       # Optional — usage examples
└── scripts/
    └── helper.py     # Optional — executable scripts
---
name: my-skill
description: Reviews code for security issues. Use when auditing code.
argument-hint: [file-or-directory]
allowed-tools: Read Grep Bash(git *)
disable-model-invocation: false
---

# My Skill

Review $ARGUMENTS for security issues.
Skill directory: ${CLAUDE_SKILL_DIR}

## Process
1. First, do this
2. Then, check that
3. Finally, verify this

Skill Invocation

Manual

Type /skill-name [args] to invoke explicitly. Arguments available via $ARGUMENTS, $0, $1.

Automatic

Claude matches your task to skill descriptions and auto-loads the relevant skill. Control with disable-model-invocation.

Programmatic

Subagents and API integrations invoke skills via the Skill tool. Claude itself can also discover and invoke built-in commands (like /init, /review, /security-review) this way. Skills can also run in isolated context with context: fork.

Auto-Invocation Tip

Front-load your description with the key use case in the first 50 characters. Claude uses this to decide when to auto-load the skill. Vague descriptions = poor triggering.

12 — Best Practices & Guidelines

Lessons from Real-World FSAD Adoption

Practical guidance for teams adopting spec-driven agentic development.

Background Context Matters

Avoid over-emphasis on technical capabilities. Insufficient business context hurts initial coding sessions. A holistic approach beyond engineering-only specifications yields best results. Include background details, business context, problems to be solved, and design systems.

Prompting

Be Specific About What, Not How

Tell Claude what outcome you want, not the implementation steps. Let the agent choose the approach based on codebase context.

Provide Business Context

Include business requirements, constraints, and existing patterns. "We need a user preferences API because mobile users complain about notification noise" is better than "add a PATCH endpoint."

Use @ References

Point Claude at specific files with @filename. This ensures the right context is loaded and prevents the agent from searching in the wrong places.

Use Plan Mode First

For complex features, start with --conversation-mode plan. Let Claude analyze and propose before making changes. Review the plan, then execute.

Project Structure

Keep CLAUDE.md Concise

Under 100 lines. If it's too long, Claude may miss critical rules. Move detailed docs to docs/ for progressive loading.

Use Local CLAUDE.md Files

Place module-specific rules near the code they protect. src/auth/CLAUDE.md gets loaded when Claude works in that directory.

Organize docs/ for Progressive Context

Architecture overview, ADRs, and runbooks in docs/. Claude reads them when relevant, not every session.

Version Control Your Prompts

Specs, skills, and configuration are code. Commit them. Review changes in PRs. Roll back if something breaks.

Workflow

Use /clear Between Tasks

Reset context between unrelated tasks. Stale context from a previous task can mislead agents on the current one.

Use /compact When Context Gets Large

Compress conversation history to free up context window space for the current task. Especially useful in long sessions.

Commit Frequently

Small, atomic changes. Each commit should represent one logical unit of work. Makes it easy to revert if an agent goes off track.

Review Agent Output Before Shipping

Agents produce consistent code from specs, but they can hallucinate or over-interpret. Human review is non-negotiable.

Common Pitfalls

Jumping Straight to Code

Without a spec, agents guess at requirements. The result is wasted cycles and code that doesn't match what was needed.

Technical Specs Without Business Context

"Add a PATCH endpoint" tells the agent what to build but not why. Missing context leads to wrong trade-offs.

Not Using Plan Mode

For complex features, jumping straight to implementation means no opportunity to catch misunderstandings before they become code.

Ignoring Context Window Limits

Overloading a session with too many files or too large a scope causes the agent to lose track of requirements mid-task.

Not Reviewing Agent Code

Agents are powerful but non-deterministic. Code that passes tests can still have logic errors, security gaps, or architectural drift.

Bloated CLAUDE.md

Stuffing every rule into CLAUDE.md defeats the purpose. Keep it focused on high-level rules; use docs/ and local CLAUDE.md for details.

Cost Optimization Checklist
  • Default to opusplan for coding — ~40% cheaper than pure Opus.
  • Route sub-agents to Haiku via CLAUDE_CODE_SUBAGENT_MODEL=haiku.
  • Enable prompt caching — cache hits are 10% of base input (up to 90% off on repeated system prompts, CLAUDE.md, large docs).
  • Use the Batch API for any non-interactive job — 50% off input and output; stacks with caching.
  • Pin effort to medium on cost-sensitive Sonnet workflows; raise only where needed.
  • Budget for up to 35% more tokens when migrating from Opus 4.6 → 4.7 (new tokenizer).

Default Effort Is Tier-Dependent

As of v2.1.117, the default effort level depends on your subscription tier and model: Pro and Max subscribers on Opus 4.6 or Sonnet 4.6 default to high. All other tiers and models default to medium. The setting is sticky — it persists across sessions. For complex agentic work, architecture decisions, and multi-file changes, use /effort max. If you've changed effort previously, verify you're on the latest Claude Code version and check that your CLAUDE.md and /memory don't have conflicting effort instructions.

Adaptive thinking & effort

On Opus 4.6 and Sonnet 4.6, adaptive thinking is the recommended mode. Instead of manually setting a thinking token budget, Claude dynamically decides when and how much to think based on task complexity. The /effort level is the primary lever that controls thinking depth:

EffortThinking Behavior
maxAlways thinks, no constraints on depth — deepest reasoning
xhighOpus 4.7 only — deeper than high, lighter than max; falls back to high on other models
highAlmost always thinks — deep reasoning on complex tasks
mediumModerate thinking — may skip for simple queries
lowMinimal thinking — skips for simple tasks, prioritizes speed

Toggle extended thinking with Option+T / Alt+T. Setting effort to low effectively minimizes thinking; use this for simple lookups or subagent tasks where speed matters most.

budget_tokens Is Deprecated

The manual budget_tokens parameter for extended thinking is deprecated on Opus 4.6 and Sonnet 4.6 and will be removed in a future release. Use adaptive thinking with the effort parameter instead. Older models (Sonnet 4.5, Opus 4.5) still require the manual budget_tokens approach.

Permission Modes

Claude asks permission for every tool use. Safest mode. Good for unfamiliar codebases.

Auto-accepts file reads and edits. Still prompts for shell commands. Good middle ground.

Read-only. Claude ONLY reads & plans. Won't write or edit files. Perfect for spec creation and analysis.

No prompts. Fastest execution. Use when you trust the agent and want maximum speed.

Hooks — Automation & Guardrails

Automate workflows, enforce policies, and extend Claude Code with event-driven hooks. Models forget. Hooks don't.

Hooks are user-defined shell commands, HTTP endpoints, LLM prompts, or agents that execute automatically at specific points in Claude Code's lifecycle. They fire deterministically — unlike instructions in CLAUDE.md that Claude may or may not follow, hooks always execute when their event triggers.

Exit Codes & Decision Control

▼

Exit codes determine whether a hook blocks execution or just logs a warning. Getting this right is critical for policy enforcement.

Exit CodeBehavior
0Success — stdout parsed for JSON output. Most events: written to debug log. Exceptions: UserPromptSubmit and SessionStart show stdout as context.
2Blocking error — blocks tool call, rejects prompt, prevents stop, or blocks compaction (depending on event). Stderr is fed back as the error message.
Any otherNon-blocking error — stderr shown as warning in transcript. Execution continues normally.
Exit 2 Is the Only Blocking Code

Exit 1 is non-blocking — it logs an error but doesn't stop anything. You must use exit 2 to enforce policy. This is the most common mistake when writing hooks.

PreToolUse permission decisions — control what happens to the tool call:

DecisionEffect
allowSkip permission prompt, execute immediately
denyBlock the tool call entirely
askPrompt the user to confirm
deferPause for later (non-interactive -p mode only)

When multiple hooks fire, precedence is: deny > defer > ask > allow.

PreCompact blocking — prevent context compaction from running:

MethodEffect
Exit with code 2Block compaction; stderr shown as reason
Return {"decision":"block"}Block compaction via JSON decision output

PostToolUse output replacement — replace the tool's output before Claude sees it:

{
  "hookSpecificOutput": {
    "updatedToolOutput": "Sanitized or transformed output here"
  }
}

Return this JSON from a PostToolUse hook (exit 0) to substitute the tool result Claude receives. Works for all tools — use it to redact sensitive output, normalize formats, or inject additional context.

▼

AGENTS.md: Project Instructions

▼

Codex merges instructions from multiple AGENTS.md files in a top-down hierarchy. Later files override earlier ones:

# Priority order (highest → lowest):
1. ~/.codex/AGENTS.override.md   # Global overrides (highest priority)
2. ~/.codex/AGENTS.md            # Global defaults
3. <repo-root>/AGENTS.md        # Project-level instructions
4. <cwd>/AGENTS.md              # Sub-directory specifics

Key differences from CLAUDE.md:

  • Override files — AGENTS.override.md takes highest priority, useful for personal preferences that override team defaults
  • Size limit — 32 KiB by default (configurable via project_doc_max_bytes in config.toml)
  • Disable flag — Skip with --no-project-doc or CODEX_DISABLE_PROJECT_DOC=1
Dual-Tool Teams

If your team uses both Claude Code and Codex, maintain both CLAUDE.md and AGENTS.md in the same repo. Keep shared conventions (tech stack, testing rules, coding standards) consistent across both files.

Running Both Tools

Many FSAD teams use both Claude Code and Codex. Keep a CLAUDE.md and AGENTS.md in your repo with shared conventions. Use each tool's strengths: Claude Code for Anthropic model quality, Codex for OpenAI model access and multi-provider flexibility. The FSAD workflow — spec → plan → implement → review — works identically regardless of which agent executes.

02 — Project Anatomy

Codex Needs 4 Things

The instructions, the skills, the guardrails, the context. Same principles as Claude — different file names.

Key Insight

Prompting is temporary. Structure is permanent. A well-organized project gives Codex context automatically — every session, every agent thread, every teammate.

Pillar 1: AGENTS.md = Repo Memory

▼

Contains WHY (purpose), WHAT (repo map), HOW (rules + commands). Codex reads this file automatically at the start of every session. The 4-level hierarchy lets you layer global defaults with project-specific overrides.

# AGENTS.md

## Project Overview
E-commerce API — Node.js + PostgreSQL. Handles product catalog,
orders, and payments.

## Tech Stack
- Runtime: Node.js 20, TypeScript
- Database: PostgreSQL 15 + Prisma ORM
- Testing: Vitest, coverage > 80%

## Commands
- `npm run dev` — start dev server
- `npm test` — run tests
- `npm run lint` — lint + format

## Rules
- Never modify migration files after they've been applied
- All API endpoints require authentication middleware
- Use Zod for request validation, not manual checks
Tip

Codex merges AGENTS.md from 4 levels: ~/.codex/AGENTS.override.md (highest) → ~/.codex/AGENTS.md → repo root → current directory. Max 32 KiB by default.

Pillar 2: .agents/skills/ = Reusable Expert Modes

▼

Turn common workflows into skills. Codex skills live in .agents/skills/ and use a SKILL.md file. Each skill can include scripts, references, and assets alongside the markdown.

# .agents/skills/code-review/SKILL.md

# Code Review

## Process
1. Check for security vulnerabilities (OWASP Top 10)
2. Verify error handling covers edge cases
3. Check test coverage for new code paths
4. Review naming conventions match project style
5. Verify no secrets or credentials in code

## Checklist
- [ ] No SQL injection vectors
- [ ] Input validation on all endpoints
- [ ] Tests cover happy path + error cases
- [ ] No console.log statements left behind

Pillar 3: .rules Files = Guardrails

▼

Codex uses the Shell Tool MCP with .rules files for deterministic guardrails — allowing, prompting, or forbidding specific shell commands. Models forget. Rules don't.

# .rules — Shell command policies

allow npm test
allow npm run lint
allow npm run build
prompt npm install *        # Ask before installing packages
forbid rm -rf *             # Never allow recursive delete
forbid git push --force     # Never allow force push
Important

Rules intercept execve(2) calls, so they know the full path of executed programs. This is more granular than Claude's hooks — you control command execution at the shell level.

Pillar 4: docs/ = Progressive Context

▼

Architecture overview, ADRs, runbooks. Don't bloat AGENTS.md — put detailed docs in files Codex can read when needed. Same pattern as Claude.

docs/
├── architecture.md        # System design overview
├── adr/                   # Architecture Decision Records
│   ├── 001-use-prisma.md
│   ├── 002-auth-strategy.md
│   └── 003-caching-layer.md
├── runbooks/              # Operational procedures
│   ├── deploy.md
│   └── incident-response.md
└── api/                   # API documentation
    └── endpoints.md

Codex discovers and reads these files on demand when working in related areas. The key is progressive context — only loaded when relevant.

Pillar 5: Local AGENTS.md for Risky Modules

▼

Place small AGENTS.md files near sharp edges. Codex sees gotchas exactly when working in that directory.

# src/auth/AGENTS.md

## Critical: Authentication Module

- NEVER modify the JWT signing key rotation logic
- Token refresh uses sliding window — do not change to fixed expiry
- Rate limiting is per-IP, configured in middleware (not here)
- All changes to this module require security review
- Test with: `npm test -- --grep "auth"`

Good candidates: src/auth/, src/billing/, src/persistence/, infra/

Setting Up MCP Connections via codex mcp

Codex uses the codex mcp CLI to manage MCP servers. For HTTP servers (Jira, Figma, Notion), use --url. For stdio servers (Azure DevOps, community tools), pass the command after --.

1

Add an MCP Server

Use codex mcp add to register a server. For hosted HTTP endpoints, use --url:

# HTTP server (hosted endpoint)
codex mcp add atlassian --url https://mcp.atlassian.com/v1/mcp

# Stdio server (local process)
codex mcp add ado -- npx -y @azure-devops/mcp your-org-name

# Stdio server with env vars
codex mcp add my-server --env API_KEY=xxx -- npx -y @some/mcp-server
2

Authenticate via OAuth

For servers that support OAuth, run codex mcp login. A browser window opens for authentication:

codex mcp login atlassian
3

Verify the Connection

Use /mcp inside the Codex TUI or codex mcp list from the terminal to confirm servers are connected:

# In the Codex TUI:
/mcp

# From the terminal:
codex mcp list
4

Manage Servers

Use codex mcp subcommands to manage your connections:

codex mcp list              # List all configured servers
codex mcp get <name>        # Show server details
codex mcp remove <name>     # Remove a server
codex mcp login <name>      # OAuth login for a server
codex mcp logout <name>     # Clear OAuth credentials
CLI vs. Config File

Use codex mcp add for quick setup — it writes to ~/.codex/config.toml automatically. Edit config.toml directly when you need advanced options like startup_timeout_sec, enabled_tools allowlists, or project-scoped servers (.codex/config.toml in trusted projects).

04 — Code Review

Built-in Code Review with /review

Codex includes a native code review command that analyzes diffs without modifying your code.

How /review Works

The /review slash command analyzes your current diff (staged and unstaged changes) and provides feedback on bugs, security issues, code quality, and style — without making any modifications to your files.

Usage

1

Start a Codex Session

Launch Codex in your project directory:

codex
2

Run /review

Type /review to analyze your current changes. Codex examines the diff and provides structured feedback:

# Review current working tree changes
/review

# Or ask for a specific review focus
"Review the changes in src/auth/ for security issues"
3

Act on Feedback

Review the findings and ask Codex to fix specific issues, or continue working on your own.

Claude Code Comparison

Claude Code uses a multi-agent fleet for code review (multiple specialized agents in parallel). Codex's /review is single-agent by default but can leverage multi-agent threads if configured. Both approaches catch bugs — Claude's is more comprehensive, Codex's is faster for quick reviews.

Example: Deployment Skill

Deploy Skill — SKILL.md

▼ 
# Deployment Skill

## Overview
Guided deployment to staging and production environments.

## Process
1. Run the test suite: `npm test`
2. Check for uncommitted changes: `git status`
3. Run the staging check script: `./scripts/check-staging.sh`
4. Build the production bundle: `npm run build`
5. Deploy to staging: `npm run deploy:staging`
6. Run smoke tests against staging
7. If smoke tests pass, deploy to production

## Safety Rules
- NEVER deploy directly to production without staging
- Always check the deploy-checklist.md reference
- Abort if any test or smoke test fails

06 — Guidelines

Lessons from Real-World Codex Adoption

Practical guidance for teams using Codex for spec-driven agentic development.

Prompting

Be Specific About What, Not How

Tell Codex what outcome you want, not the implementation steps. Let the agent choose the approach based on codebase context.

Provide Business Context

Include business requirements, constraints, and existing patterns. "We need a user preferences API because mobile users complain about notification noise" is better than "add a PATCH endpoint."

Use Tab for File References

Press Tab to fuzzy-search files by name (prefix with @). This ensures the right context is loaded and prevents Codex from searching in the wrong places.

Use /plan Mode First

For complex features, start with /plan. Let Codex analyze and propose before making changes. Review the plan, then let it execute.

Project Structure

Keep AGENTS.md Focused

Stay under the 32 KiB limit. Put detailed docs in docs/ for progressive context — don't bloat your project instructions file.

Use Skills for Repeatable Workflows

If you find yourself giving the same instructions across sessions, encode them as a skill in .agents/skills/.

Use Local AGENTS.md for Sensitive Modules

Place small AGENTS.md files near auth, billing, and infrastructure code. Codex sees the guardrails exactly when working in that directory.

Use /compact to Manage Context

Long conversations eat context. Use /compact to summarize the conversation and free up tokens for the agent to work with.

Anti-Patterns to Avoid

Over-Permissive Sandbox

Using danger-full-access outside of containers defeats the security model. Stick with workspace-write for development.

Bloated AGENTS.md

Stuffing every rule into AGENTS.md defeats the purpose. Keep it focused on high-level rules; use docs/ and local AGENTS.md for details.

Ignoring Session Resume

Codex saves sessions automatically. Use codex resume to pick up where you left off instead of re-explaining context from scratch.

Not Using /compact

Long conversations degrade agent performance. Use /compact regularly to summarize and free context tokens.

Multi-Agent Workflows

▼

Codex supports experimental multi-agent threads with configurable concurrency and specialized roles.

# config.toml
[agents]
max_threads = 6
max_depth = 1
job_max_runtime_seconds = 3600

Built-in roles:

  • default — Standard agent behavior
  • worker — Focused task execution
  • explorer — Read-only codebase analysis
  • monitor — Long-running observation (up to 1-hour windows)

Batch processing: spawn_agents_on_csv reads a CSV and spawns one worker per row — useful for bulk migrations and parallel test generation.

▼

01 — FSAD Metrics

Measuring FSAD Success

What to measure to demonstrate the value of agentic development to your organization.

Highest Priority

Token Consumption by User

Track per-developer AI usage to identify adoption patterns and cost centers. High consumption with high output signals effective use; high consumption with low output signals coaching opportunities.

Total Monthly Epics / Tickets

Volume of work items completed, comparing pre- vs. post-FSAD adoption. Expect a meaningful increase as AI agents accelerate implementation cycles.

Total Monthly Story Points

Throughput measurement for sprint-over-sprint comparison. Track velocity trends after FSAD adoption to quantify productivity gains.

Total PRs / Commits

Development velocity indicator. AI-assisted workflows typically produce higher commit frequency with smaller, more focused changesets.

Maintenance vs. New Product Development

Breakdown of time and code produced by maintenance vs. new product development. Measures whether AI frees engineering capacity for innovation over sustaining work.

Total $ Spend by AI Tool

Cost tracking per tool (Claude Code, Codex, Cursor, etc.) for ROI analysis. Correlate spend against velocity and output metrics to demonstrate return on investment.

Other Metrics

AI Coding Tools WAU

Weekly Active Users across AI coding tools. Measures adoption breadth across the engineering organization.

Lines of Code Produced

AI-generated vs. human-written code split. Track the ratio over time to understand how AI contribution evolves.

AI Commits vs. Acceptance Rates

How often AI-generated code is accepted vs. revised. A quality signal — high acceptance rates indicate well-configured agents and effective specs.

Start Small

Don't instrument everything on day one. Token consumption + story points alone can demonstrate ROI within a single sprint. Add granularity as your FSAD practice matures and leadership asks deeper questions.
