01 — What is FSAD

The Paradigm Shift

FSAD replaces the traditional handoff-heavy SDLC with a tight feedback loop where humans define what and why in markdown — and AI agents handle the how.

🎯

Spec-Driven, Not Code-Driven

The specification — written in markdown — is the primary artifact. It's a living document that evolves in real-time during collaborative sessions. Code is a compilation target, not the source of truth.

🤝

Cross-Functional Pods

Small teams of 2–4 (PM + UX + Engineer, or PM + 2 Engineers) sit together — physically or virtually — and co-author the spec in real-time. No handoffs. No waiting for the next sprint.

📝

Markdown as Common Language

Markdown is lightweight, version-controlled, diff-friendly, and — critically — the native instruction format for AI agents. Every role can read and write it. It bridges the gap between human intent and machine execution.

🤖

Agent-Delegated Execution

Once the spec is validated, teams of AI agents — running in parallel via tools like Claude Code, Codex, or Gemini CLI — implement the code. The human role shifts from writing code to reviewing, steering, and verifying agent output.

Phase 1 — Intent & Discovery

The PM and UX designer define the problem space and user outcomes. They collaborate — often with an AI research assistant — to surface user needs, competitive landscape, and constraints. The output is a concise problem statement in markdown that frames what we're solving and for whom. No solutions yet.

problem.mduser-stories.mdconstraints.md

Phase 2 — Live Spec Session

The entire pod convenes in a real-time collaborative session. The PM contributes business requirements, UX contributes interaction patterns and design decisions, and the Engineer contributes technical feasibility and architecture. They co-author the spec in markdown together — design decisions are embedded inline, not deferred to a separate Figma-to-Jira handoff. This is where FSAD breaks the handoff chain.

spec.mddesign-decisions.mdAGENTS.md

Phase 3 — Spec Validation & Task Decomposition

The Engineer feeds the spec to an AI agent in "plan mode" (read-only). The agent analyzes the codebase, generates an implementation plan, and decomposes work into discrete, parallelizable tasks. The human reviews the plan against the spec — catching hallucinations, missed edge cases, and architectural misalignment before any code is written.

plan.mdtasks/*.mdtest-plan.md

Phase 4 — Parallel Agent Execution

Multiple AI agents execute tasks simultaneously in isolated git worktrees. Each agent works against the spec as its source of truth. A Coordinator agent (or human) routes tasks, and a Verifier agent checks each output against the spec's acceptance criteria. The engineer monitors, steers, and intervenes only when agents get stuck or drift.

src/**tests/**agent-logs/

Phase 5 — Human Review & Ship

The full pod reviews the implementation. The Engineer checks code quality, architecture adherence, and test coverage. UX validates the interface against design decisions in the spec. The PM validates against user stories and acceptance criteria. Reviews focus on spec compliance, not syntax — because agents produce consistent code from living specs. Then ship.

The Markdown is the Meeting

In FSAD, the spec.md file replaces most of the ceremony of traditional development — the Jira tickets, the Figma handoffs, the sprint planning debates, the slack threads asking "what did we decide?"

AI Agents consume the spec and execute implementation in parallel — the "workforce" that the pod directs.

03 — Pod Anti-Patterns

Compositions That Don't Work

🚫

The Solo Hero

One engineer doing everything. No second perspective means spec blind spots go unchallenged. Agents amplify mistakes at scale.

🚫

The Oversized Pod

5+ humans on a single spec. Communication overhead grows quadratically. Split into two pods with a shared API contract.

🚫

The Handoff Pod

PM writes spec → throws to designer → throws to engineer. Same sequential process. If you aren't co-authoring, it's not FSAD.

🚫

The Agent-Only Pod

Fully autonomous agents, no human review. Agent reliability is non-deterministic. Every pod requires human validation gates.

🚫

The Part-Time Pod

Members at 20% allocation across multiple pods. FSAD's speed comes from real-time collaboration. Async stalls the spec session.

🚫

The Duplicate Pod

Two engineers with identical skills. The second must own a distinct domain — otherwise use a Triad and let agents parallelize.

The Principle

Smallest Effective Pod.
Maximum Agent Leverage.

The right pod is the smallest group of humans who can make complete decisions about a feature without external dependencies — then direct an arbitrarily large fleet of agents to execute in parallel.

▼

A real-world CLAUDE.md covering workflow orchestration, task management, and core principles. Copy and adapt for your team.

## Workflow Orchestration

### 1. Plan Mode Default
- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately — don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

### 2. Subagent Strategy
- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

### 3. Self-Improvement Loop
- After ANY correction from the user: update `tasks/lessons.md` with the pattern
- Write rules for yourself that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops
- Review lessons at session start for relevant project

### 4. Verification Before Done
- Never mark a task complete without proving it works
- Diff behavior between main and your changes when relevant
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness

### 5. Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes — don't over-engineer
- Challenge your own work before presenting it

### 6. Autonomous Bug Fixing
- When given a bug report: just fix it. Don't ask for hand-holding
- Point at logs, errors, failing tests — then resolve them
- Zero context switching required from the user
- Go fix failing CI tests without being told how

## Task Management

1. **Plan First**: Write plan to `tasks/todo.md` with checkable items
2. **Verify Plan**: Check in before starting implementation
3. **Track Progress**: Mark items complete as you go
4. **Explain Changes**: High-level summary at each step
5. **Document Results**: Add review section to `tasks/todo.md`
6. **Capture Lessons**: Update `tasks/lessons.md` after corrections

## Core Principles

- **Simplicity First**: Make every change as simple as possible. Impact minimal code.
- **No Laziness**: Find root causes. No temporary fixes. Senior developer standards.
- **Minimal Impact**: Changes should only touch what's necessary. Avoid introducing bugs.

10.6 — Security Review

Multi-Agent Security Review Team

Orchestrate a team of specialized review agents in parallel for a pre-merge or pre-release hardening pass. Each specialist writes findings to its own file; a lead agent consolidates them into a single report. Review only — no fixes.

The Specialists

  • auth-authz-auditor — Auth flows, sessions, JWT/OAuth, RBAC, privilege boundaries, endpoint auth mapping.
  • input-validation-auditor — Injection vectors: SQLi, command injection, SSRF, path traversal, XSS, deserialization, prompt injection.
  • secrets-crypto-auditor — Hardcoded secrets, weak crypto, insecure random, improper key/token storage, leaked credentials.
  • dependency-supplychain-auditor — Known CVEs, pinned versions, lockfile integrity, suspicious packages, CI/CD trust boundaries.
  • silent-failure-hunter — Swallowed exceptions, fallbacks that mask security failures, error handling that fails open.
  • data-exposure-auditor — PII handling, log redaction, response shapes, IDOR, tenant isolation, overly broad CORS.

Example Prompt

Paste this into Claude Code, replace <PATH> and the scope placeholder, and adjust the agent roster to match your stack:

Spin up a security review team for the codebase at <PATH>. Scope: <all backend services | specific dirs | recent diff vs main>.

1. Create a team via TeamCreate named "sec-review" with a task list.

2. Spawn these specialists in parallel via the Agent tool, each with one focused task and writing findings to .planning/security-review/<agent>.md:

   - **auth-authz-auditor** (security-auditor): Review all auth flows, session handling, JWT/OAuth, RBAC, and privilege boundaries. Map every endpoint to its auth requirement.

   - **input-validation-auditor** (pr-review-toolkit:code-reviewer): Hunt injection vectors — SQLi, command injection, SSRF, path traversal, XSS, deserialization, prompt injection in LLM calls. Trace every external input to its sink.

   - **secrets-crypto-auditor** (feature-dev:code-reviewer): Find hardcoded secrets, weak crypto, insecure random, improper key/token storage, logged credentials, secrets in error messages.

   - **dependency-supplychain-auditor** (general-purpose): Audit dependencies for known CVEs (pip-audit / npm audit), pinned versions, lockfile integrity, suspicious packages, and CI/CD trust boundaries.

   - **silent-failure-hunter** (pr-review-toolkit:silent-failure-hunter): Catch swallowed exceptions, fallback paths that mask security failures, and error handling that fails open instead of closed.

   - **data-exposure-auditor** (code-reviewer): PII handling, log redaction, response shapes that leak internals, IDOR, missing tenant isolation, overly broad CORS.

3. Each agent must:
   - Read only files in scope
   - Produce findings as: [SEVERITY: critical/high/medium/low] file:line — issue — exploit scenario — recommended fix
   - Cite concrete evidence; no speculation
   - Mark its task complete via TaskUpdate when done

4. After all agents finish, you (lead) consolidate into .planning/security-review/REPORT.md grouped by severity, dedupe overlaps, and propose a fix order. Then shut down the team.

Use the code-review skill to frame the consolidation. Do NOT fix anything in this pass — review only.
How to Adapt

Swap in your stack's auditors (e.g., drop dependency-supplychain-auditor for a pure static site, add a Terraform/IaC auditor for infra repos). Scope tightly — a diff-only review finishes in minutes; a full-repo sweep can take much longer and cost more.

12 — Best Practices & Guidelines

Lessons from Real-World FSAD Adoption

Practical guidance for teams adopting spec-driven agentic development.

Background Context Matters

Avoid over-emphasis on technical capabilities. Insufficient business context hurts initial coding sessions. A holistic approach beyond engineering-only specifications yields best results. Include background details, business context, problems to be solved, and design systems.

Prompting

Be Specific About What, Not How

Tell Claude what outcome you want, not the implementation steps. Let the agent choose the approach based on codebase context.

Provide Business Context

Include business requirements, constraints, and existing patterns. "We need a user preferences API because mobile users complain about notification noise" is better than "add a PATCH endpoint."

Use @ References

Point Claude at specific files with @filename. This ensures the right context is loaded and prevents the agent from searching in the wrong places.

Use Plan Mode First

For complex features, start with --conversation-mode plan. Let Claude analyze and propose before making changes. Review the plan, then execute.

Project Structure

Keep CLAUDE.md Concise

Under 100 lines. If it's too long, Claude may miss critical rules. Move detailed docs to docs/ for progressive loading.

Use Local CLAUDE.md Files

Place module-specific rules near the code they protect. src/auth/CLAUDE.md gets loaded when Claude works in that directory.

Organize docs/ for Progressive Context

Architecture overview, ADRs, and runbooks in docs/. Claude reads them when relevant, not every session.

Version Control Your Prompts

Specs, skills, and configuration are code. Commit them. Review changes in PRs. Roll back if something breaks.

Workflow

Use /clear Between Tasks

Reset context between unrelated tasks. Stale context from a previous task can mislead agents on the current one.

Use /compact When Context Gets Large

Compress conversation history to free up context window space for the current task. Especially useful in long sessions.

Commit Frequently

Small, atomic changes. Each commit should represent one logical unit of work. Makes it easy to revert if an agent goes off track.

Review Agent Output Before Shipping

Agents produce consistent code from specs, but they can hallucinate or over-interpret. Human review is non-negotiable.

Common Pitfalls

Jumping Straight to Code

Without a spec, agents guess at requirements. The result is wasted cycles and code that doesn't match what was needed.

Technical Specs Without Business Context

"Add a PATCH endpoint" tells the agent what to build but not why. Missing context leads to wrong trade-offs.

Not Using Plan Mode

For complex features, jumping straight to implementation means no opportunity to catch misunderstandings before they become code.

Ignoring Context Window Limits

Overloading a session with too many files or too large a scope causes the agent to lose track of requirements mid-task.

Not Reviewing Agent Code

Agents are powerful but non-deterministic. Code that passes tests can still have logic errors, security gaps, or architectural drift.

Bloated CLAUDE.md

Stuffing every rule into CLAUDE.md defeats the purpose. Keep it focused on high-level rules; use docs/ and local CLAUDE.md for details.

02 — Project Anatomy

Codex Needs 4 Things

The instructions, the skills, the guardrails, the context. Same principles as Claude — different file names.

Key Insight

Prompting is temporary. Structure is permanent. A well-organized project gives Codex context automatically — every session, every agent thread, every teammate.

Pillar 1: AGENTS.md = Repo Memory

▼

Contains WHY (purpose), WHAT (repo map), HOW (rules + commands). Codex reads this file automatically at the start of every session. The 4-level hierarchy lets you layer global defaults with project-specific overrides.

# AGENTS.md

## Project Overview
E-commerce API — Node.js + PostgreSQL. Handles product catalog,
orders, and payments.

- Runtime: Node.js 20, TypeScript
- Database: PostgreSQL 15 + Prisma ORM
- Testing: Vitest, coverage > 80%

## Commands
- `npm run dev` — start dev server
- `npm test` — run tests
- `npm run lint` — lint + format

## Rules
- Never modify migration files after they've been applied
- All API endpoints require authentication middleware
- Use Zod for request validation, not manual checks
Tip

Codex merges AGENTS.md from 4 levels: ~/.codex/AGENTS.override.md (highest) → ~/.codex/AGENTS.md → repo root → current directory. Max 32 KiB by default.

Pillar 2: .agents/skills/ = Reusable Expert Modes

▼

Turn common workflows into skills. Codex skills live in .agents/skills/ and use a SKILL.md file. Each skill can include scripts, references, and assets alongside the markdown.

- [ ] No SQL injection vectors

- [ ] No console.log statements left behind

Pillar 3: .rules Files = Guardrails

▼

Codex uses the Shell Tool MCP with .rules files for deterministic guardrails — allowing, prompting, or forbidding specific shell commands. Models forget. Rules don't.

# .rules — Shell command policies

allow npm test
allow npm run lint
allow npm run build
prompt npm install *        # Ask before installing packages
forbid rm -rf *             # Never allow recursive delete
forbid git push --force     # Never allow force push
Important

Rules intercept execve(2) calls, so they know the full path of executed programs. This is more granular than Claude's hooks — you control command execution at the shell level.

Pillar 4: docs/ = Progressive Context

▼

Architecture overview, ADRs, runbooks. Don't bloat AGENTS.md — put detailed docs in files Codex can read when needed. Same pattern as Claude.

Prompting

Be Specific About What, Not How

Tell Codex what outcome you want, not the implementation steps. Let the agent choose the approach based on codebase context.

Provide Business Context

Include business requirements, constraints, and existing patterns. "We need a user preferences API because mobile users complain about notification noise" is better than "add a PATCH endpoint."

For complex features, start with /plan. Let Codex analyze and propose before making changes. Review the plan, then let it execute.

If you find yourself giving the same instructions across sessions, encode them as a skill in .agents/skills/.

Using danger-full-access outside of containers defeats the security model. Stick with workspace-write for development.

Codex saves sessions automatically. Use codex resume to pick up where you left off instead of re-explaining context from scratch.

Not Using /compact

Long conversations degrade agent performance. Use /compact regularly to summarize and free context tokens.

01 — FSAD Metrics

Measuring FSAD Success

What to measure to demonstrate the value of agentic development to your organization.

Highest Priority

Token Consumption by User

Track per-developer AI usage to identify adoption patterns and cost centers. High consumption with high output signals effective use; high consumption with low output signals coaching opportunities.

Total Monthly Epics / Tickets

Volume of work items completed, comparing pre- vs. post-FSAD adoption. Expect a meaningful increase as AI agents accelerate implementation cycles.

Total Monthly Story Points

Throughput measurement for sprint-over-sprint comparison. Track velocity trends after FSAD adoption to quantify productivity gains.

Total PRs / Commits

Development velocity indicator. AI-assisted workflows typically produce higher commit frequency with smaller, more focused changesets.

Maintenance vs. New Product Development

Breakdown of time and code produced by maintenance vs. new product development. Measures whether AI frees engineering capacity for innovation over sustaining work.

Total $ Spend by AI Tool

Cost tracking per tool (Claude Code, Codex, Cursor, etc.) for ROI analysis. Correlate spend against velocity and output metrics to demonstrate return on investment.

Other Metrics

AI Coding Tools WAU

Weekly Active Users across AI coding tools. Measures adoption breadth across the engineering organization.

Lines of Code Produced

AI-generated vs. human-written code split. Track the ratio over time to understand how AI contribution evolves.

AI Commits vs. Acceptance Rates

How often AI-generated code is accepted vs. revised. A quality signal — high acceptance rates indicate well-configured agents and effective specs.

Start Small

Don't instrument everything on day one. Token consumption + story points alone can demonstrate ROI within a single sprint. Add granularity as your FSAD practice matures and leadership asks deeper questions.
