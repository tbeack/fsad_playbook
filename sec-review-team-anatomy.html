sec-review-team

A multi-agent security review skill. Orchestrates a roster of specialist agents in parallel — each scanning a distinct threat domain — then consolidates findings into a single severity-ranked report.

⚠ REVIEW ONLY — This skill never edits files, applies fixes, or opens PRs. Fix workflow → companion skill /sec-review-fixes

Deference
data-exposure → frontend-security
When frontend-security-auditor is in the roster, data-exposure defers CSP, CORS, SRI, and cookie flags. If frontend-security is absent, data-exposure reclaims those categories as fallback.
Deference
data-exposure → privacy-telemetry
When privacy-telemetry-auditor is in the roster, data-exposure defers analytics SDKs, consent flow, and cross-border data transfer. Data-exposure stays on what could leak; privacy-telemetry owns what deliberately leaves.
Deference
input-validation → prompt-injection
When prompt-injection-auditor is in the roster, input-validation marks LLM input sinks as deferred-to-other-specialist in coverage.jsonl and leaves those paths to the prompt-injection specialist.
Cross-cut
silent-failure ↔ auth-authz
auth-authz owns architectural IPC/capability issues; silent-failure owns error-path silences in those checks. Both coordinate on findings where an auth check exists but its failure is swallowed.
Cross-cut
container-runtime ↔ iac
iac-auditor owns cluster-level resources (IAM, network policies, Helm chart values). container-runtime owns pod-level security context (runAsRoot, capabilities.add, readOnlyRootFilesystem). K8s manifests sit at the boundary — both specialists coordinate.
Cross-cut
ci-cd ↔ dependency-supplychain
ci-cd owns workflow triggers and action pinning. dependency-supplychain owns package-dep security. Both cross-cut on action-version supply chain (third-party GitHub Actions) — ci-cd owns the pinning concern, deps owns CVE-in-action.

Workflow — Step by Step

Each step in detail, including the safety gates and orchestrator contracts.

0
Pre-run Confirmation Gate
The largest user-facing safety gate. Never spawns specialists without user approval (unless --yes is passed).
  • Parse target path (absolute) and scope (all / subdir / diff vs main)
  • Enumerate files + lines via git ls-files or find. Cap at 500k lines — warn and suggest narrowing if exceeded.
  • Detect stack signals by reading manifests (package.json, Cargo.toml, Dockerfile, etc.)
  • Apply stack-signals.md rules → union specialist sets → default roster
  • Check scanner availability via command -v for each of 16 scanners
  • Estimate tokens / runtime / cost using CBP-060 baseline rates
  • Render confirmation block. User can: y, narrow, add-specialist, drop-specialist, or abort
Abort exits cleanly — no directory created, no agents spawned.
1
Final Roster + Run Metadata
Write run-metadata.json capturing run_id, target, scope, stack signals, roster, scanner status, model, and cost estimate. Load specialist briefs only for confirmed roster members.
  • Consumed by Step 4 consolidation (fills REPORT.md header)
  • Enables re-run comparison across time (CBP-066)
  • Specialist briefs loaded lazily — no briefs for dropped specialists
2
Prepare Findings Directory
Create <TARGET>/.planning/security-review/ if absent. Check .gitignore for pollution risk.
  • If .planning/ is NOT gitignored, warn and offer: (a) add to .gitignore, (b) write to /tmp/sec-review-<ts>/, or (c) proceed
  • Findings co-located with code — enables history across re-runs
2.5
Scanner Pre-Pass (parallel)
Deterministic scanners run before LLM specialists. This is the biggest accuracy-per-token lever — specialists spend tokens on signal tools can't catch.
  • Run available scanners in parallel (wall-clock ceiling: 120s per scanner)
  • Normalize all outputs to finding.schema.json format
  • Append to scanner-prepass.jsonl with source: "scanner-<name>"
  • Feed relevant findings to each specialist as "Pre-pass context" in their brief
  • Specialists triage: true-positive / false-positive / needs-investigation
  • Zero scanners available → run continues in LLM-only mode (CBP-060 baseline)
3
Spawn Specialists in Parallel
All specialist agents are issued in a single assistant message — this is what makes them run concurrently.
  • Each specialist gets: brief, substituted target/scope/stack-context, pre-pass findings for their domain
  • Tool allowlists enforced at harness level: Read, Grep, Glob, read-only Bash, Write scoped to 4 output paths only
  • Each specialist writes heartbeat <name>.status.json at start, during scan (every ~5 reads), and on completion/error
  • Orchestrator prints one status line as each agent returns: ✓ auth-authz done (0C/2H/2M/1L) [4m22s]
  • Errored specialists don't block the run — partial findings preserved
Monitor live: watch -n 5 'jq -s "sort_by(.agent)" .planning/security-review/*.status.json'
4
Consolidation
Reads all *.findings.jsonl and *.coverage.jsonl, deduplicates, and renders REPORT.md.
  • Group by root_issue slug → merge: max(severity), max(confidence), distinct raised-by list
  • Split: actionable (certain/likely) vs worth-investigating (possible/unverified)
  • Build coverage matrix: category × specialist table with status + confidence cells
  • REPORT.md includes source column so scanner-confirmed vs specialist-found findings are visible
  • Scanner-only findings (specialist errored before triage) flagged ⚠ unreviewed by specialist
5
Deliver
Present findings path, headline counts, top 2–3 fixes by impact, and tooling caveats. No fixes applied.
  • Path to REPORT.md and per-specialist files
  • Deduped severity counts (critical / high / medium / low)
  • Top priority fixes by blast radius
  • Coverage completeness score
  • Tooling caveats: which scanners unavailable, which specialists errored

Finding Schema

Each specialist emits one JSON object per line in <name>.findings.jsonl, validated against finding.schema.json.

id required
string (slug: ^[a-z0-9-]+$)
Stable per-run finding ID. Prefix with specialist short code, e.g. auth-001.
specialist required
enum (13 values)
Which of the 13 registered specialists emitted this finding.
severity required
critical | high | medium | low | info
On merge, max severity across group wins. critical > high > medium > low > info.
confidence required
certain | likely | possible | unverified
certain = directly observed + mechanical fix. likely = pattern + inference. possible = indirect evidence. unverified = can't confirm without runtime.
root_issue required
string (slug)
Short dedupe key across specialists. Same root_issue = merge into one report entry, e.g. csp-null, unscoped-sql-capability.
title required
string (≤ 140 chars)
Human-readable finding title for REPORT.md headings.
file required
string (repo-relative path)
Repo-relative path to the affected file. Used for deduplication and coverage matrix.
exploit required
string
Concrete attack scenario. Synthesized across group in REPORT.md.
fix required
string
Specific change to make. Scanner-sourced findings may leave this blank; specialist fills on triage.
evidence required
string
Exact code snippet or config block, quoted verbatim. No paraphrase.
source
string (default: "specialist")
Provenance: specialist, scanner-<name>, scanner-<name>-confirmed, or scanner-<name>-false-positive.
line_range
array [start, end]
Line range in the affected file. 1-indexed.
threat_model
network-attacker | insider | supply-chain | disk-access | physical | llm-adversarial-input | n/a
Primary adversarial model this finding applies to.
cwe / cve / tags
string / string / array
Optional enrichment. CWE in CWE-NNN format. CVE in CVE-YYYY-NNN format.
related
array of finding IDs
Cross-specialist correlation. IDs of findings from other specialists that touch the same scope. Used to build cross-reference section in REPORT.md.

Consolidation Rules

Same root_issue → merge
Findings sharing a root_issue slug are merged into one REPORT.md entry with max(severity), max(confidence), and a union of raised_by specialists.
Same location, different root_issue → keep separate
Findings at the same file:line_range but different root_issue remain separate — they're distinct issues co-located in the same code.
Actionable vs worth-investigating
certain/likely confidence → Action items. possible/unverified → Worth investigating. REPORT.md uses two separate sections.

Deliberate Non-Features & Tradeoffs

Design decisions that look like gaps but are intentional. Captured so future contributors know the reasoning.

Tradeoff 1
general-purpose fallback is the intended path
Each specialist lists a preferred_subagent_type (e.g., security-auditor) with fallback: general-purpose. When the preferred type isn't registered, the orchestrator falls back silently.
Validated on Opus 4.7 (CBP-060) — findings quality didn't degrade when all 6 specialists ran as general-purpose. The brief is what produces focus, not the agent type.
Tradeoff 2
Findings land inside the target repo
Output goes to <TARGET>/.planning/security-review/. The skill checks .gitignore for pollution risk and offers /tmp/ as an alternative.
Preserves history across runs, enables re-run comparison, co-locates findings with the code they describe. Target-repo owner already has access to findings.
Tradeoff 3
No streaming during specialist runs
The Agent tool is synchronous — the orchestrator blocks until a subagent returns. Specialists write .status.json heartbeats for live monitoring via tail -f or watch in a second terminal.
Platform limitation. Status-file dance is the workaround until Agent tool gains streaming semantics.
Tradeoff 4
Review-only is enforced by tool allowlists
Prompt-level "don't edit" is a wish. Tool allowlists passed via allowed-tools restrict each specialist to Read, Grep, Glob, and Write scoped to the 4 output paths. Edit and non-listed Bash are denied at the harness level.
Hard enforcement boundary — a misbehaving or prompt-injected agent cannot silently mutate the target codebase.
Tradeoff 5
No automatic fix application — ever
Fix-application risk is fundamentally different from review risk. Fix workflow is a separate companion skill: /sec-review-fixes (CBP-070), which consumes REPORT.md and proposes fix PRs with interactive approval gates.
Separate skill = separate sandbox = separate approval surface. This separation is load-bearing and intentional. Not a gap to fill.
Tradeoff 6
Two copies of the canonical prompt
The original monolithic prompt lives at markdown/design/Team_of_security_agents.md. The playbook embeds a copy in the Security Review HTML section. Sync is manual.
CBP-071 refactors the playbook section to render a summary + invocation example pointing at the skill, removing the inline full-prompt duplication.

What this skill deliberately is NOT

✗ A replacement for human security review — it's a force-multiplier
✗ A CI gate for new vulns — that's a scanner's job
✗ A fix-generating tool — that's /sec-review-fixes
✗ A compliance audit — regulatory mapping is out of scope
✗ A pentest — no dynamic analysis, no exploitation, no fuzzing
